Escape Vector
textures/Escapevector.png #ff0000

Escape-Vector is a retro-styled 2.5D roguelike shooter where 2D characters are running through a 3D world full of neon chaos. Made this for WhalePass Jam #2 and the theme was "Retro" — so I went full 80s neon shooter vibes.

Media:

Tech:

  • Unity (3D) – Game Engine
  • C# – Programming
  • Blender – For 3D buildings and props

What I Did:

The main goal of the jam was to use the WhalePass API — so I integrated it to track XP, unlock achievements, and handle upgrades based on the player’s level. Basically, it's got online progression baked right in.

Now for the visuals — I went for that old-school CRT arcade look. I baked a bunch of lighting directly into textures using Unity, then removed the actual lights from the scene and swapped in emissive maps. That way I got fake-but-fresh lighting with ambient occlusion, while keeping the performance smooth.

To really push the retro style, I used Unity's custom render pipeline and made a post-processing shader that pixelates only specific objects — so the characters and bullets look crunchy, while the environment stays crisp.

Shoutout to one of the cool folks from the jam’s Discord — they helped with the soundtrack, and the music totally slaps with the retro action vibe.

This jam was a 7-day sprint, so it was hectic — but I learned a ton about shader tricks, API integration, and squeezing style into performance.

SYSRA.asm

SYSRA.asm is an 8086 Assembly-language platformer prototype I developed to explore low-level game development on DOS. It demonstrates custom sprite rendering, collision detection, and direct screen-buffer management—all handcrafted without any external libraries.

Tech:

  • NASM / 8086 real-mode Assembly – full control over CPU instructions.
  • BIOS & DOS interrupts – setting video mode (0x13h), keyboard input, timing.
  • Direct video memory writes – drawing sprites and tiles in real-time.
  • Fixed-point arithmetic – subpixel movement and smooth collision handling.
  • Double buffering in segment:offset schema – to reduce flickering.
  • Custom tile & sprite system – collision map integrated into binary formats.

What I Did

I built this from scratch to understand how classic DOS games managed graphics and input. I implemented:

  • Real-mode memory segmentation and segment:offset pointers for tile maps.
  • An update-render loop that manually toggles between backbuffer and frontbuffer.
  • A fixed-point system (8.8 format) for smooth horizontal and vertical movement.
  • Collision detection by checking sprite footprint against a binary collision mask.
  • Keyboard polling via BIOS INT 16h to get immediate player input.
  • Optimizations like unrolling inner loops and using REGISTER-level sprite blit routines.

Through this project, I gained deep insights into:

  • CPU pipeline and instruction latency on 8086.
  • Memory layout constraints in limited segment offsets.
  • Manual collision system design at the byte-level.
  • Graphics mode 0x13h (320×200, 256‑color) rendering challenges.

Though not a full game yet, it forms the foundation for a retro-style platformer engine. I intend to extend it with sound support (via PC speaker), level loading from data files, and rudimentary physics.

Check out the full source and demos on GitHub

2DURP‑GodRays
Video/2DURP.mp4 #ffffcc

2DURP‑GodRays is a post-processing shader crafted for Unity’s URP that generates dynamic, high‑quality god ray (crepuscular rays) effects in 2D scenes. It supports customizable settings like density, decay, exposure, and blur, enabling both aesthetic and optimized performance integration. A demo is available on itch.io.

Tech:

  • Unity Universal Render Pipeline (URP) – for lightweight scriptable rendering.
  • HLSL-based custom full-screen post-processing shader.
  • Render Features & Render Pass – injecting god‑ray pass after depth/opaque.
  • Depth-based radial blur – sample along vector from light source, combine decay.
  • Shader properties: Density, Decay, Exposure, Samples, BlurRadius.
  • Support for both 2D Sprite & 2D Lights pipelines via URP layering.
  • Efficient performance modes: adjustable sample count and resolution via downsampling.

What I Did

  • Architected a URP ScriptableRendererFeature and ScriptableRenderPass for light scattering.
  • Worked with RTHandles and temporary render textures to isolate and blur light source masks.
  • Developed radial blur kernel in HLSL with decay control and multi-iteration sampling.
  • Exposed shader tuning parameters, enabling artists/devs to tweak visuals at runtime.
  • Handled performance via dynamic sample count scaling and viewport downscaling.
  • Created demonstration scene (sprites, parallax background) with UI toggles and keybindings: U, A/D/Arrows, Space, LMB/RMB, F :contentReference[oaicite:3]{index=3}.

What I Learned

  • Deep understanding of URP’s render lifecycle and passes.
  • Techniques for efficient radial blur and light-scattering shader design.
  • Memory management using RTHandles and transient RenderTextures.
  • Balancing visual quality and performance through sampling strategies.
  • Shader‑centered UI configuration for rapid iteration and testing.

This shader pack is a foundation for visual polish in 2D URP games. Future improvements include:

  • Dynamic light occlusion via depth masks or stencil overlays.
  • Auto light positioning from scene light sources (e.g., 2D Light and Shadow Caster).
  • Integration with volumetric particle overlays for atmosphere enhancement.
  • Support for HDR and bloom integration via post-processing stack.

Physics‑Engine
Video/PhysicsEngine.mp4 #ffffcc

Physics‑Engine is a custom 2D physics simulation framework written in C++ with OpenGL rendering. It demonstrates core physics techniques like Verlet integration, constraint-based springs and distance joints, and simple collision detection/rendering—all built from the ground up.

Tech:

  • C++11 – core architecture and physics logic.
  • GLFW + GLEW – cross-platform OpenGL context creation and function loading.
  • OpenGL – real-time rendering using point and line primitives for particles and joints.
  • Verlet Integration – stable time-stepping for particle position updates.
  • Constraint Solvers – spring/distance constraints implemented via positional corrections.
  • Basic collision detection – AABB/circle or point collision responses.
  • User input – interactive control via keyboard (Space for jump, Up arrow to move).
  • FPS locking and loop timing – fixed at 144 Hz for deterministic updates.

What I Did

  • Implemented a particle-based physics system using Verlet integration to simulate stable motion.
  • Built constraint structures: springs and fixed-distance joints connecting particles.
  • Developed simple collision detection and response to prevent particles/joints from passing through static obstacles.
  • Rendered simulation elements (particles and joints) using OpenGL’s line and point drawings.
  • Integrated real-time input handling: players can apply impulses (e.g. jump) via Space and steer using arrow keys.
  • Ensured smooth physics with 144 Hz loop, locking FPS for consistent behavior across runs.

What I Learned

  • Low-level physics integration methods and their stability at high update rates.
  • Constraint resolution techniques using positional corrections instead of force solvers.
  • Real-time rendering with OpenGL primitives and handling graphics context with GLFW/GLEW.
  • Loop timing and deterministic simulation by decoupling physics updates from frame rendering.
  • Interactive input mapping to physics actions for user-controlled simulation scenarios.

This project is a foundation for advanced physics-based systems. Future extensions could include:

  • Rigid-body dynamics with orientation/quaternion handling.
  • More advanced collision detection (SAT, GJK) and continuous collision response.
  • Soft-body structures like cloth or mesh deformation.
  • Adding textures, shaders for visualization, and performance profiling tools.

See the full demo (video linked in README) and source code on GitHub.
