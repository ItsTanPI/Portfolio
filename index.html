Escape Vector
textures/Escapevector.png #ff0000

Escape-Vector is a retro-styled 2.5D roguelike shooter where 2D characters are running through a 3D world full of neon chaos. Made this for WhalePass Jam #2 and the theme was "Retro" — so I went full 80s neon shooter vibes.

Media:

Tech:

  • Unity (3D) – Game Engine
  • C# – Programming
  • Blender – For 3D buildings and props

What I Did:

The main goal of the jam was to use the WhalePass API — so I integrated it to track XP, unlock achievements, and handle upgrades based on the player’s level. Basically, it's got online progression baked right in.

Now for the visuals — I went for that old-school CRT arcade look. I baked a bunch of lighting directly into textures using Unity, then removed the actual lights from the scene and swapped in emissive maps. That way I got fake-but-fresh lighting with ambient occlusion, while keeping the performance smooth.

To really push the retro style, I used Unity's custom render pipeline and made a post-processing shader that pixelates only specific objects — so the characters and bullets look crunchy, while the environment stays crisp.

Shoutout to one of the cool folks from the jam’s Discord — they helped with the soundtrack, and the music totally slaps with the retro action vibe.

This jam was a 7-day sprint, so it was hectic — but I learned a ton about shader tricks, API integration, and squeezing style into performance.

PiGL.js
textures/pigl.png #ff00aa

PiGL.js is a custom WebGL rendering engine and editor built from scratch — vanilla JS and GLSL, no heavy frameworks. Modular architecture, multipass rendering, deferred lighting, post-processing effects. More of a graphics playground than a game engine, and that's exactly the point.

Tech:

  • Vanilla JavaScript (ES6+) – core engine logic, OOP architecture, async asset loading
  • WebGL 1.0 / GLSL – raw graphics, lighting, and screen-space effects
  • Vite – build tooling, shader hot-reloading
  • lil-gui – in-editor UI for real-time inspection and material tweaking

What I Did

  • Engineered a modular rendering pipeline from scratch — Lighting, Outlines, Skybox, Pixel Art passes
  • Built a deferred-like workflow using MRTs to decouple lighting from scene complexity
  • Created an in-browser editor: Window Manager, Hierarchy view, real-time Inspector
  • Implemented a full math library — custom Matrix4, Vector3, Quaternion-style logic
  • Built an OBJ + texture asset loader with placeholder and GPU upload state management
  • Designed a real-time profiler for FPS, frame times, and per-pass draw call counts

What I Learned

  • Low-level WebGL state management — framebuffers, vertex/fragment shaders, the whole pipeline
  • Designing extensible engine architecture where adding a new visual pass doesn't break everything else
  • Spatial math for camera systems, projection matrices, and light-space transforms
  • Optimizing render loops through resource reuse and minimizing state changes

Powers stylized pixel-art environments and water simulations. Source and docs on GitHub.

SYSRA.asm

SYSRA.asm is SYSRA — but in 8086 Assembly, running on DOS. Custom sprite rendering, collision detection, direct screen-buffer management. No libraries, no engine. Just registers, interrupts, and a deep understanding of how old games actually worked.

Tech:

  • NASM / 8086 real-mode Assembly – full CPU instruction control
  • BIOS & DOS interrupts – video mode (0x13h), keyboard input, timing
  • Direct video memory writes – real-time sprite and tile rendering
  • Fixed-point arithmetic (8.8 format) – subpixel movement and smooth collision
  • Double buffering in segment:offset – flickering eliminated
  • Custom tile & sprite system – collision map baked into binary format

What I Did

  • Real-mode memory segmentation and segment:offset pointers for tile maps
  • Manual update-render loop toggling between backbuffer and frontbuffer
  • Fixed-point system for smooth horizontal and vertical movement
  • Collision detection via sprite footprint checked against a binary collision mask
  • Keyboard polling via BIOS INT 16h for immediate player input
  • Inner loop unrolling and REGISTER-level sprite blit optimizations

What I Learned

  • CPU pipeline and instruction latency at the 8086 level
  • Memory layout constraints in limited segment offsets
  • Manual collision system design at the byte level
  • Graphics mode 0x13h (320×200, 256-color) rendering constraints and tricks

Foundation for a retro-style platformer engine. Planning to extend with PC speaker sound, level loading, and rudimentary physics. Full source on GitHub.

2DURP‑GodRays
Video/2DURP.mp4 #ffffcc

2DURP‑GodRays is a post-processing shader for Unity URP that generates dynamic god ray (crepuscular ray) effects in 2D scenes. Fully tunable — Density, Decay, Exposure, Blur, Samples. Demo on itch.io.

Tech:

  • Unity URP – lightweight scriptable rendering
  • HLSL – custom full-screen post-processing shader
  • ScriptableRendererFeature + ScriptableRenderPass – injected after depth/opaque pass
  • Depth-based radial blur – samples along vector from light source with decay
  • RTHandles – transient render texture management
  • Dynamic sample scaling and viewport downscaling for performance modes

What I Did

  • Developed the radial blur kernel in HLSL with multi-iteration decay sampling
  • Managed RTHandles and temporary render textures to isolate and blur light source masks
  • Exposed all shader parameters for runtime tweaking by artists and devs
  • Built a demo scene with UI toggles and keybinds for live testing

What I Learned

  • URP's render lifecycle and the full ScriptableRenderPass workflow
  • Radial blur and light-scattering shader design techniques
  • RTHandle memory management and transient render texture best practices
  • Balancing visual quality vs performance through sampling strategies

Physics‑Engine
Video/PhysicsEngine.mp4 #ffffcc

Physics‑Engine is a custom 2D physics simulation in C++ with OpenGL rendering. Verlet integration, constraint-based springs and distance joints, basic collision — all from scratch. Built it because I wanted to understand constraint solvers before trusting any engine's physics black box.

Tech:

  • C++11 – core architecture and physics logic
  • GLFW + GLEW – cross-platform OpenGL context and function loading
  • OpenGL – real-time rendering with point and line primitives
  • Verlet Integration – stable time-stepping for particle position updates
  • Constraint Solvers – spring/distance constraints via positional corrections
  • 144 Hz fixed loop – deterministic physics updates

What I Did

  • Particle-based physics system using Verlet integration for stable motion
  • Spring and fixed-distance joint constraints connecting particles
  • Basic collision detection and response for static obstacles
  • Real-time OpenGL rendering of particles and joints via line/point primitives
  • Interactive input — Space for impulse/jump, arrow keys to steer

What I Learned

  • Physics integration methods and their stability characteristics at high update rates
  • Constraint resolution via positional corrections instead of force-based solvers
  • Real-time rendering with OpenGL primitives and GLFW/GLEW context management
  • Decoupling physics updates from frame rendering for deterministic simulation

Full source and demo on GitHub.
